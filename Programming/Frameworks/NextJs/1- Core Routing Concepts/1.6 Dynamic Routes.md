# Dynamic Routes
Pull your chair back over. We’ve mastered static routing—creating specific folders for specific URLs. If you want an /about page, you make an about folder. That works perfectly for a marketing site.
But let's think about scale. What if you are building an e-commerce site with 10,000 different sneakers? Or a blog with 500 articles? You cannot manually create 10,000 folders and paste a page.tsx into every single one. You'd lose your mind, and the codebase would be unmaintainable. We need a way to generate pages dynamically based on the URL.
## The "Why": The Need for Templates
When a user visits your-site.com/products/air-jordan and another visits your-site.com/products/yeezy, the overall UI on the screen is exactly the same. There is a giant image on the left, a title on the right, and an "Add to Cart" button at the bottom.
The only thing that changes is the *data* injected into that UI.
We need a way to tell Next.js: *"Hey, if a user goes to /products/ANYTHING, catch that request, load up my single product template, and tell me what the 'ANYTHING' was so I can fetch the right data from the database."*
## The Mental Model: Fill-in-the-Blanks
Think of Dynamic Routes like a game of Mad Libs, or a nametag at a conference.
The nametag comes pre-printed with: Hello, my name is ______. The blank space is a placeholder. It doesn't care if your name is Sarah, John, or Rumpelstiltskin. It just holds the space.
In Next.js, we create these blank spaces using **square brackets**. By naming a folder with brackets—like [slug] or [id]—we are telling the framework that this is a dynamic placeholder, not a literal word.
## Concept Before Syntax: Matching the Path
Let's look at how the router reads these brackets.
If you create this folder structure: app/blog/[slug]/page.tsx
Next.js will match all of these URLs to that exact same page.tsx file:
 * /blog/hello-world
 * /blog/my-first-post
 * /blog/12345
But how does your React component actually know which URL the user typed? Next.js automatically takes the exact text from the URL and passes it into your page component as a **prop** called params.
## Building Incrementally: The Code
Let's see what that looks like in practice. Whatever word you put inside the square brackets for the folder name becomes the key in your params object.
```tsx
// File: app/blog/[slug]/page.tsx

export default function BlogPostPage({ params }: { params: { slug: string } }) {
  // If the user visited /blog/hello-world, params.slug will equal "hello-world"
  
  return (
    <main>
      <h1>You are reading: {params.slug}</h1>
      <p>Now we can use this slug to fetch the real article from our database!</p>
    </main>
  );
}

```
*Note: In the absolute newest versions of Next.js (version 15+), params is technically a Promise that you have to await. But the core pattern and mental model remain exactly the same: you get the data from the URL, and you use it to render the page.*
## Teaching Patterns: Static vs. Dynamic Priority
Here is a brilliant pattern Next.js handles for you behind the scenes: **Static routes always win.**
Imagine you have a dynamic route for users: app/users/[username]/page.tsx. This catches /users/john and /users/sarah.
But what if you also need a general settings page at /users/settings? You might worry that "settings" will accidentally get swallowed by the [username] dynamic folder.
Don't panic. If you explicitly create a static folder next to it (app/users/settings/page.tsx), Next.js is smart enough to check for exact static matches *before* falling back to the dynamic bracket folder. The static route always takes priority.
## Mistakes to Avoid Before You Make Them
 * **Mistake 1: Mismatching the folder name and the prop name.**
   * *Reality:* If you name your folder [id], but in your code you try to read params.slug, it will be undefined. The name inside the brackets dictates the object key. They must match perfectly.
 * **Mistake 2: Assuming the dynamic segment is always a string.**
   * *Reality:* Even if your URL is /products/123, the params.id you get in your code will be the string "123", not the number 123. If your database needs an integer, you have to parse it manually (parseInt(params.id)).
 * **Mistake 3: Trying to make the page.tsx itself dynamic.**
   * *Reality:* Remember the rule from day one: the file must literally be named page.tsx. You apply the brackets to the *folder* ([slug]/page.tsx), never to the file itself ([slug].tsx).
## Practical Exercise
Let's build a dynamic route in our sandbox right now.
 1. Open your app directory.
 2. Create a new folder named products.
 3. Inside products, create another folder, and physically type the brackets in the name: [productId].
 4. Inside [productId], create a page.tsx file.
 5. Add this simple component:
   ```tsx
   export default function ProductPage({ params }: { params: { productId: string } }) {
     return <h1>Viewing Product ID: {params.productId}</h1>;
   }
   
   ```
 6. Open your browser and go to http://localhost:3000/products/shoes. You'll see "Viewing Product ID: shoes".
 7. Change the URL to http://localhost:3000/products/999. You'll instantly see "Viewing Product ID: 999".
You just built a single file that can handle an infinite number of URLs. That is the power of dynamic routing.
