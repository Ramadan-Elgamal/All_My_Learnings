# Pages and Layouts
Pull up your chair. We know how to create individual URLs using folders and page.tsx. But if you think about any real-world application, pages rarely exist in isolation. You usually have a navigation bar at the top, a footer at the bottom, or maybe a sidebar. Let's talk about how we share that UI without repeating ourselves.
## Start with the "Why": The Re-rendering Problem
In a standard React application, sharing UI like a Navbar across different routes can be messy. You either import <Navbar /> into every single page component—which is tedious and prone to errors—or you wrap your entire router in a giant layout component.
But there's a hidden performance cost in traditional React routing: when you navigate from the Home page to the About page, React often destroys and rebuilds the entire component tree, including that shared Navbar. If your Navbar had complex state (like a dropdown menu or a search input), that state gets wiped out on navigation.
Next.js solves this elegantly by introducing a clear boundary between **Layouts** and **Pages**.
## The Mental Model: The Picture Frame and the Photograph
Think of your application like a picture frame hanging on a wall.
 * **The Layout (layout.tsx) is the picture frame.** It provides the structure, the borders, and the matting. Once you hang it on the wall, it stays exactly where it is.
 * **The Page (page.tsx) is the photograph inside the frame.** This is the unique, specific content.
When you navigate to a new URL, Next.js doesn't take the frame off the wall. It just carefully slides out the old photograph and slides in the new one. The frame (Layout) never re-renders, which means any state inside it is completely preserved.
## Concept Before Syntax: Wrapping with children
To make this "picture frame" concept work in code, Next.js relies heavily on a core React concept: the children prop.
Whenever Next.js sees a layout.tsx file inside a folder, it renders that layout first. Then, it looks for the page.tsx file in that same folder (or any nested sub-folders), takes that page's UI, and injects it directly into the layout's children prop.
This pattern is called **Nesting Layouts**. The root layout wraps everything. A folder inside it can have its own layout, which wraps its specific pages, creating a Russian nesting doll of user interfaces.
## Building Incrementally: Writing a Layout
Let's look at what a basic layout actually looks like. Notice how it takes in children and wraps them in a <div> that acts as our sidebar structure.
```tsx
// app/dashboard/layout.tsx
export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <section style={{ display: 'flex' }}>
      {/* This sidebar stays put */}
      <nav>
        <ul>
          <li>Dashboard Home</li>
          <li>Settings</li>
        </ul>
      </nav>
      
      {/* The specific page content gets injected here */}
      <main>{children}</main>
    </section>
  );
}

```
## Common Mistakes Before You Make Them
 * **Mistake 1: Forgetting to accept or render children.**
   * *Reality:* If you write a layout.tsx but forget to include {children} in the return statement, your layout will render, but the actual page content will completely disappear. Next.js has nowhere to inject the "photograph."
 * **Mistake 2: Expecting Layouts to re-render on navigation.**
   * *Reality:* Because layouts preserve state, useEffect hooks inside a layout won't re-trigger just because the user clicked a link to a new page within that same layout. If you *want* a layout to completely rebuild on every navigation (for example, to trigger a page-transition animation), you should use a different Next.js file called template.tsx.
 * **Mistake 3: Putting layout logic in page.tsx.**
   * *Reality:* Don't import your Navbar into your individual pages. Trust the file system. Put the Navbar in your layout, and let Next.js handle the composition.
## Practical Exercise
Let's build out a dashboard layout to see this state preservation in action. Make sure your dev server is running.
 1. In your app directory, create a new folder called admin.
 2. Inside admin, create a layout.tsx file.
 3. Paste the following code into it:
   ```tsx
   'use client'; // We will explain this in a future lesson, just add it for now!
   import { useState } from 'react';
   
   export default function AdminLayout({ children }: { children: React.ReactNode }) {
     const [count, setCount] = useState(0);
   
     return (
       <div style={{ border: '2px solid blue', padding: '20px' }}>
         <h2>Admin Sidebar</h2>
         <button onClick={() => setCount(count + 1)}>Clicks: {count}</button>
         <div style={{ marginTop: '20px', border: '1px solid gray', padding: '10px' }}>
           {children}
         </div>
       </div>
     );
   }
   
   ```
 4. Inside the admin folder, create a page.tsx file that just returns <h1>Admin Dashboard</h1>.
 5. Inside the admin folder, create a sub-folder called users, and put a page.tsx in it that returns <h1>User Management</h1>.
 6. Open http://localhost:3000/admin in your browser. Click the button a few times so the counter goes up.
 7. Now, change the URL to http://localhost:3000/admin/users.
 8. Watch closely: the inner content changed to "User Management," but the blue border stayed, and more importantly, **your click count did not reset to zero**.
That is the power of Next.js layouts. They wrap your pages and protect your state.
