# Templates (Next.js)
Grab a coffee. In the last file, we looked at how Layouts wrap your pages and tightly hold onto their state as users navigate around. That state preservation is amazing for performance and user experience most of the time.
But what if you *don't* want the state to stick around?
Sometimes, you actually want the UI to completely rebuild from scratch when the user clicks a link. That is exactly where template.tsx comes in. It is the Layout's slightly more destructive sibling.
## The Mental Model: The Etch-a-Sketch
If a Layout is a sturdy picture frame that stays bolted to the wall while the photo inside changes, a **Template is an Etch-a-Sketch**.
Every time a user navigates to a new page that shares the template, Next.js shakes the Etch-a-Sketch. It completely destroys the old component, throws away any state tied to it, and draws a brand-new version of the UI from scratch.
Behind the scenes, Next.js does this by giving the template component a unique key prop based on the current route. In React, when a component's key changes, React forcibly unmounts the old component and mounts a new one.
## When Do You Actually Use a Template?
Since Layouts are much more efficient (because they don't rebuild the DOM), you should default to using layout.tsx 95% of the time. You only reach for a template.tsx when you have a specific reason to force a re-render.
Here are the most common real-world use cases:
**1. Enter/Exit Animations**
If you are using a library like Framer Motion to make a page fade in or slide across the screen, a Layout won't work. The Layout never unmounts, so the animation only plays once when the user first arrives. A Template rebuilds on every navigation, so your animate-in effect will trigger beautifully every single time the user clicks a link.
**2. Resetting State on Navigation**
Imagine a custom feedback form or a "give us a rating" widget that sits at the bottom of every blog post. If you put that in a Layout, a user could type "Great post!" on Post A, click the link to Post B, and "Great post!" would still be sitting in the text box. By putting the widget in a Template, the form's local state automatically wipes clean when they switch posts.
**3. Relying on useEffect**
If you have a hook that *must* fire on every single page view (like a custom page-view analytics tracker), a Template ensures that effect re-runs because the component is freshly mounted.
## How It Looks in Code
Syntax-wise, a Template looks entirely identical to a Layout. It exports a default React component and accepts children.
```tsx
// app/template.tsx
'use client';

import { motion } from 'framer-motion';

export default function RootTemplate({ children }: { children: React.ReactNode }) {
  // Because this is a template, this animation will run 
  // every single time the route changes.
  return (
    <motion.div
      initial={{ opacity: 0, y: 20 }}
      animate={{ opacity: 1, y: 0 }}
      transition={{ duration: 0.5 }}
    >
      {children}
    </motion.div>
  );
}

```
## How They Behave Together
You don't have to choose between them; you can use both in the exact same folder. When you colocate a layout.tsx and a template.tsx, Next.js wraps them in a specific order:
 1. First, it renders the Layout (which stays static).
 2. Inside the Layout, it renders the Template (which rebuilds).
 3. Inside the Template, it renders the Page.
So, your persistent Sidebar stays in the Layout, while your fading page transition lives in the Template right next to it.
## A Quick Warning
Don't use a Template just because you are struggling with state bugs in a Layout. If you find yourself reaching for template.tsx because your Layout's state is getting tangled, take a step back. Usually, that means you need to lift your state higher or use a URL query parameter, rather than taking the heavy-handed approach of forcing React to nuke the entire component tree on every click.
## Practical Exercise
Let's see the Etch-a-Sketch in action.
 1. Go back to the admin folder we built in the last file.
 2. Next to your layout.tsx, create a file named template.tsx.
 3. Add this code to track how many times the component actually mounts:
```tsx
'use client';
import { useEffect } from 'react';

export default function AdminTemplate({ children }: { children: React.ReactNode }) {
  useEffect(() => {
    console.log("Template Mounted: The route just changed!");
  }, []);

  return (
    <div style={{ border: '2px dashed red', padding: '10px' }}>
      <p>I am the Template</p>
      {children}
    </div>
  );
}

```
 4. Open your browser console.
 5. Click back and forth between /admin and /admin/users.
 6. You will see the blue border (Layout) stay perfectly still, the click counter state persist, but the red dashed border (Template) will physically flash, and your console will log the mount message every single time.
That's the fundamental difference. Use Layouts for persistence, and Templates for a fresh start.
