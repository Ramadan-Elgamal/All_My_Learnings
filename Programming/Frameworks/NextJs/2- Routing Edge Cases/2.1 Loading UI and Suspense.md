# Loading UI and Suspense

So far, our pages have loaded instantly because they only return static HTML.

Real applications are different.

Pages often need to:

- Query databases
- Fetch data from APIs
- Read files
- Perform expensive server-side operations

All of these take time.

If we don't handle that delay properly, the application will feel slow and unresponsive.

---

# The Frozen Screen Problem

Imagine a user clicks the **Billing History** link.

The request reaches the Next.js server, which starts querying the database.

Suppose the query takes **3 seconds**.

Without a loading state:

1. The user clicks the link.
2. Nothing changes.
3. The user thinks the app is broken.
4. They click the link again.
5. Three seconds later, the new page suddenly appears.

This creates a poor user experience.

Instead, we want immediate feedback.

As soon as the user navigates, the layout should remain visible while the page displays a temporary loading interface until the data arrives.

---

# Mental Model: Building Skeleton First

Imagine a skyscraper under construction.

The workers don't finish the entire building somewhere else and then move it into place.

Instead, they first build the steel framework.

Even before the walls and furniture exist, everyone can already see the building taking shape.

Next.js follows the same idea.

It first sends a lightweight placeholder—the **skeleton**—and then fills in the real content as soon as the data becomes available.

---

# React Suspense

The feature that makes this possible is **React Suspense**.

Suspense acts like a safety net around components that may take time to render.

It works like this:

```text
Component starts loading
          │
          ▼
Is it ready?
      │
 ┌────┴────┐
 │         │
No        Yes
 │         │
 ▼         ▼
Show    Render
Fallback Component
```

A Suspense boundary has two parts:

- The component that might suspend.
- A fallback UI displayed while it loads.

Once the component finishes loading, React automatically replaces the fallback with the real content.

---

# `loading.tsx`

Writing Suspense boundaries everywhere would become repetitive.

Next.js simplifies this with a special file:

```text
loading.tsx
```

Whenever this file exists inside a route folder, Next.js automatically wraps that route in a Suspense boundary.

The `loading.tsx` component becomes the fallback UI.

Example:

```tsx
// app/dashboard/loading.tsx

export default function DashboardLoading() {
  return (
    <div className="skeleton-wrapper">
      <h2>Loading your dashboard...</h2>
      <div className="spinner"></div>
    </div>
  );
}
```

When a user visits `/dashboard`:

1. The layout renders immediately.
2. `loading.tsx` appears instantly.
3. The server fetches the required data.
4. The actual page replaces the loading UI automatically.

---

# Why Layouts Matter

Remember the rendering hierarchy:

```text
layout.tsx
    │
    ▼
loading.tsx
    │
    ▼
page.tsx
```

Because the loading UI is rendered **inside** the layout:

- Navigation remains visible.
- Sidebar remains interactive.
- Header doesn't disappear.
- Only the page content is temporarily replaced.

The application never feels frozen.

---

# The Limitation of `loading.tsx`

`loading.tsx` works well for entire pages.

However, it has one limitation.

Imagine a dashboard containing:

- User Profile → loads in **0.1 seconds**
- Revenue Chart → loads in **4 seconds**

With a single `loading.tsx`:

```text
Loading...

(wait 4 seconds)

Everything appears at once
```

The fast component is forced to wait for the slow one.

This is unnecessary.

---

# Streaming with Granular Suspense

A better approach is to wrap only the slow components in their own Suspense boundaries.

```tsx
import { Suspense } from "react";

import FastProfileWidget from "./FastProfileWidget";
import SlowRevenueChart from "./SlowRevenueChart";

export default function DashboardPage() {
  return (
    <main>
      <h1>Dashboard</h1>

      <FastProfileWidget />

      <Suspense fallback={<p>Loading chart data...</p>}>
        <SlowRevenueChart />
      </Suspense>
    </main>
  );
}
```

Now the rendering looks like this:

```text
Dashboard Title
User Profile
Loading chart...

↓

A few seconds later...

Dashboard Title
User Profile
Revenue Chart
```

Only the slow component waits.

Everything else becomes usable immediately.

This technique is called **Streaming**.

---

# `loading.tsx` vs Manual Suspense

|`loading.tsx`|Manual `Suspense`|
|---|---|
|Wraps the entire route automatically|Wraps only selected components|
|Great for simple pages|Great for complex dashboards|
|One loading UI for everything|Different loading UIs for different sections|
|Easy to set up|More flexible|

---

# Common Mistakes

## 1. Expecting `loading.tsx` to Work with Client-Side Fetching

`loading.tsx` only works with **Server Components** that suspend while rendering.

It does **not** activate for data fetched inside a Client Component using `useEffect`.

### Doesn't trigger `loading.tsx`

```tsx
"use client";

useEffect(() => {
  fetch("/api/products");
}, []);
```

### Works with `loading.tsx`

```tsx
export default async function Page() {
  const data = await fetch(...);

  return (...);
}
```

---

## 2. Placing `loading.tsx` at the Root

```
app/
├── loading.tsx
```

This causes **every navigation** in your application to display the loading UI.

Instead, place loading files only where they're needed.

Example:

```
app/
└── products/
    └── [productId]/
        ├── page.tsx
        └── loading.tsx
```

Now only product pages show that loading state.

---

## 3. Fetching Data Inside `loading.tsx`

`loading.tsx` should never contain business logic.

Its only responsibility is displaying placeholder UI.

### Good

```tsx
export default function Loading() {
  return <Spinner />;
}
```

### Avoid

```tsx
export default async function Loading() {
  await fetch(...);

  return <Spinner />;
}
```

All data fetching belongs in:

- `page.tsx`
- Server Components
- Child components
