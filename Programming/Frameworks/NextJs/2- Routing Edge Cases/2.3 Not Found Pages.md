# Not Found Pages
Pull your chair back up.
We just learned how to handle broken code using error.tsx.
But there is another type of failure we need to manage.
What happens when the code works perfectly, but the *thing* the user is looking for just isn't there?
A bad URL.
A deleted blog post.
An expired product.
If we let these fall into our error.tsx file, the user sees a red "Something went wrong!" screen.
But nothing went wrong. The system worked. The item is just missing.
We need a dedicated way to say, "We looked, but we couldn't find it."
## The "Why": The 404 vs 500 Difference
In web development, HTTP status codes matter.
A 500 error means "Server Crash." That is what error.tsx is for.
A 404 error means "Not Found."
Search engines care about this distinction.
If Google tries to crawl a deleted blog post and gets a 500 error, it assumes your site is broken and might penalize your rankings.
If it gets a 404, it correctly understands the page is gone and removes it from the search index.
## Mental Model: The Empty Shelf
Think of a library.
If the roof collapses, that is an error.tsx. Nobody gets any books.
If you walk in, ask the librarian for a specific book, and they check the system and say, "Sorry, someone checked that out," that is a not-found.tsx.
The library is fine. The shelf is just empty.
Next.js gives us a specific file to design that "empty shelf" experience.
## The not-found.tsx File
Just like loading.tsx and error.tsx, Next.js uses file-system conventions for 404 pages.
If you create a file named not-found.tsx inside a folder, it handles missing resources for that specific route segment.
```tsx
// app/not-found.tsx
import Link from 'next/link';

export default function GlobalNotFound() {
  return (
    <div className="not-found-container">
      <h2>404 - Page Not Found</h2>
      <p>We couldn't find the page you were looking for.</p>
      <Link href="/">Return to Homepage</Link>
    </div>
  );
}

```
Notice that unlike error.tsx, a not-found.tsx file does not need the "use client" directive. It can be a standard Server Component.
## Teaching Patterns: The Two Ways a 404 Happens
In Next.js, a Not Found UI is triggered in two completely different ways.
**Trigger 1: Unmatched URLs (The framework handles it)**
If a user types your-site.com/dashboard/magic-unicorn-page.
Next.js looks at your app folder.
It doesn't see a magic-unicorn-page folder.
It automatically renders the closest not-found.tsx file it can find.
**Trigger 2: Missing Data (You handle it manually)**
If a user goes to your-site.com/products/123.
Next.js *does* find a match! It matches the dynamic route app/products/[productId]/page.tsx.
The framework thinks everything is fine.
But then, your component checks the database for Product 123. The database says, "That product was deleted."
Next.js doesn't know that. You have to tell it.
## The notFound() Function
To manually trigger the 404 UI when data is missing, we use a special function provided by Next.js.
It is called notFound().
```tsx
import { notFound } from 'next/navigation';
import db from '@/lib/db';

export default async function ProductPage({ params }: { params: { productId: string } }) {
  
  // 1. Fetch the data
  const product = await db.getProduct(params.productId);

  // 2. If no data comes back, trigger the 404 page manually
  if (!product) {
    notFound(); 
  }

  // 3. If we get here, the product exists. Render the UI.
  return <h1>{product.name}</h1>;
}

```
## How notFound() Actually Works
Here is a critical concept to grasp:
notFound() is not a standard return statement.
Under the hood, notFound() actually throws a special JavaScript error.
The moment you call notFound(), the execution of your component stops immediately.
Any code written below notFound() will never run.
Next.js catches that special error and instantly swaps the screen to your not-found.tsx file.
## Common Mistakes
**1. Forgetting to return or stop execution**
Because notFound() throws an error, you don't need to write return notFound().
Just calling notFound() is enough to stop the page dead in its tracks.
**2. Trying to catch notFound() in error.tsx**
Even though notFound() throws an error under the hood, Next.js is smart enough to bypass your error.tsx boundary.
It specifically routes to the not-found.tsx file. They do not conflict.
**3. Relying only on the root 404**
You can have multiple not-found.tsx files.
You might want a generic one at app/not-found.tsx for bad URLs.
But you might want a specific one at app/products/not-found.tsx that says: "We couldn't find this specific product. Here are some other shoes you might like."
Keep your 404s contextual.
## Practical Exercise
Let's test manual data 404s in our sandbox.
**Step 1**
Go to your app/products/[productId] folder.
**Step 2**
Create a new file:
not-found.tsx
**Step 3**
Add your empty shelf UI:
```tsx
import Link from 'next/link';

export default function ProductNotFound() {
  return (
    <div style={{ padding: "20px", border: "2px solid orange" }}>
      <h2>Product Missing</h2>
      <p>We searched the warehouse, but this product doesn't exist.</p>
      <Link href="/products">Go back to all products</Link>
    </div>
  );
}

```
**Step 4**
Open your page.tsx in that same folder.
Import the notFound function at the top, and add a fake database check:
```tsx
import { notFound } from 'next/navigation';

export default async function ProductPage({ params }: { params: { productId: string } }) {
  
  // Simulate checking a database
  // If the user types '999', we pretend it was deleted.
  if (params.productId === '999') {
    notFound();
  }

  return <h1>Viewing Product ID: {params.productId}</h1>;
}

```
**Step 5**
Start the development server.
Visit http://localhost:3000/products/123. You should see the standard product page.
Now, visit http://localhost:3000/products/999.
Execution immediately stops, and your custom, orange "Product Missing" UI renders instead.
