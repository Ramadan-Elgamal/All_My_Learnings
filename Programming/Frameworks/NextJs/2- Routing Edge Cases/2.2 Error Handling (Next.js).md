# Error Handling (Next.js)
Grab your chair.
We just learned how to handle slow pages.
Now we need to handle broken ones.
Because in the real world, things break.
APIs go down.
Databases time out.
We write bad code.
If we don't prepare for this, the user experience is catastrophic.
## The White Screen of Death Problem
In standard React, an unhandled error is fatal.
If a single component throws an error, React unmounts the entire component tree.
The user is left staring at a completely blank white screen.
They don't know what happened.
They don't know how to fix it.
They just leave.
We cannot let one broken button or one failed database query take down the entire application.
## Mental Model: The Blast Door
Think of a submarine.
If a leak happens in one compartment, they don't let the whole submarine sink.
They seal the blast doors to that specific room.
The room is flooded, but the rest of the submarine keeps functioning.
In Next.js, we use **Error Boundaries** as our blast doors.
When a page crashes, we contain the explosion to that specific part of the screen.
The layout, the sidebar, and the navigation all stay perfectly intact.
## React Error Boundaries and error.tsx
React has a built-in feature called Error Boundaries to catch these crashes.
But writing them manually requires complex class components.
Next.js automates this entirely.
You just create a file named:
error.tsx
Whenever this file exists in a folder, Next.js automatically wraps that route in a React Error Boundary.
If the page crashes, Next.js instantly replaces the broken page with your error.tsx component.
## Writing an Error Component
An error.tsx file must be a Client Component.
This is because errors can happen on the server *or* in the browser. A Client Component can catch both.
Next.js passes two important props to this file:
 1. error: The actual error object.
 2. reset: A function to try re-rendering the page.
Here is what it looks like:
```tsx
// app/dashboard/error.tsx
"use client"; // Always required for error boundaries

export default function DashboardError({
  error,
  reset,
}: {
  error: Error;
  reset: () => void;
}) {
  return (
    <div className="error-container">
      <h2>Something went wrong!</h2>
      <p>{error.message}</p>
      
      {/* Give the user a way to recover */}
      <button onClick={() => reset()}>Try again</button>
    </div>
  );
}

```
## The "Try Again" Button
The reset() function is incredibly powerful.
Usually, errors are temporary (like a network blip).
When the user clicks "Try again", Next.js attempts to re-render the original page content.
If it works this time, the error UI disappears, and the real page comes back.
## How Errors Bubble Up
Errors always bubble *up* to the nearest blast door.
If you have this structure:
```text
app/
├── error.tsx
└── dashboard/
    ├── invoices/
    │   └── page.tsx (Crashes!)
    └── error.tsx

```
The error in invoices/page.tsx will be caught by dashboard/error.tsx.
It looks for the closest error.tsx moving up the folder tree.
## The Layout Catch
There is one critical rule about layouts and errors.
An error.tsx file **cannot** catch errors in a layout.tsx file within the exact same folder.
This is because of how Next.js nests them:
```text
layout.tsx
│
└── ▼ error.tsx (The Blast Door)
    │
    └── ▼ page.tsx

```
The blast door is *inside* the layout.
If the layout itself crashes, the door can't save it. The error will bubble up to the parent folder's error.tsx.
If your root app/layout.tsx crashes, you need a special file called global-error.tsx to catch it.
## Common Mistakes
**1. Forgetting "use client"**
error.tsx must have "use client" at the very top.
If you forget it, the error boundary itself will crash, causing a double-error.
**2. Trying to catch 404s**
error.tsx is for unexpected exceptions and crashes.
If a user visits a URL that doesn't exist, or a database record isn't found, that is a different state.
Do not use error.tsx for that. We will use not-found.tsx (covered in the next file).
**3. Leaking sensitive data**
The error.message might contain sensitive database table names or API keys if it originated on the server.
Next.js strips out server-side error messages in production for security. It sends a generic hash instead.
Don't rely on error.message being perfectly readable to the user in production. Write your own friendly fallback text.
## Practical Exercise
Let's break our application on purpose.
**Step 1**
Go to your app/products/[productId] folder.
**Step 2**
Create a new file:
error.tsx
**Step 3**
Add the blast door UI:
```tsx
"use client";

export default function ProductError({ error, reset }: { error: Error; reset: () => void }) {
  return (
    <div style={{ padding: "20px", border: "2px solid red" }}>
      <h2>Failed to load product.</h2>
      <p>Error: {error.message}</p>
      <button onClick={() => reset()}>Retry</button>
    </div>
  );
}

```
**Step 4**
Open your page.tsx in that same folder.
Let's add a landmine. Before the return statement, add this code:
```tsx
// Generate a random number between 0 and 1
const random = Math.random();

// 50% chance to crash
if (random > 0.5) {
  throw new Error("Database connection lost!");
}

```
**Step 5**
Start your development server.
Visit http://localhost:3000/products/123
**Step 6**
Refresh the page a few times.
Half the time, the page will load perfectly.
Half the time, it will crash.
When it crashes, you will see your red Error Boundary. The whole app didn't die—just that specific component.
Click the "Retry" button to trigger the reset() function and watch it recover.
